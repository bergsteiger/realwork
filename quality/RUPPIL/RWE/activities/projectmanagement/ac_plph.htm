Оценить проект

Цель
  • Оценить объемы работ
  • Выбрать оптимальное расписание, удовлетворяющее ограничениям проекта

В течение Фазы Начало, проводится оценка работ по проекту (об этом см. [BOE81], [PUT92] и [MCO96]). Оценка программного проекта основана на математических выкладках и здесь не приводится, она проводится в 4 стадии:

  1. Оценка размера продукта.
  2. Оценка общей трудоемкости и цены
  3. Привязка ограничений и приоритетов (например, численность состава, дата поставки, бюджет)
  4. Выбор оптимального расписания, трудозатрат и оценка стоимости

Оценка размера продукта

Это ключевая информация для оценки процесса. Если вы не сможете оценить сложность выполняемых работ, расписание проекта будет далеко от реальности. Многие сведения для оценки размера продукта выясняются из Видения. Есть 2 подхода к оценке размера программного продукта в начале проекта: Измерение по Аналогии и Измерение по Анализу. Позднее (на фазе Уточнение) можно подготовить более строгие оценки.

Измерение по Аналогии

Этот вид оценки основан на сравнении производимого продукта с ранее созданными. Вы должны сравнить различные характеристики сравниваемого продукта, например, количество бизнес-функций, количество актеров,размер и сложность Базы Данных, число запускаемых и работающих приложений.

Сравнивая эти характеристики, вы можете приблизительно оценить, зная величины обоих проектов, трудоемкость текущего. Однако в данной оценке нужно подбирать проекты, похожие по сложности.

Измерение по Анализу

Когда собрано достаточно информации по новому продукту, можно использовать уже аналитическую технику для оценки размеров проекта. Эта техника зависит от функционального описания программного продукта (например, Спецификация требований к программному продукту, Документ программная архитектура) и использует стандартные правила подсчета для определения размера исходя из этих описаний. Самая известная техника - это Подсчет Функциональных точек, хотя число других разработанных измерений включает Точки Возможностей (Feature Points) - модификация метода Функциональных точек для систем реального времени, а также Точки Предсказываемых объектов (основанная на анализе классов, их сложности и иерархии методология для объектно-ориентированных систем)

Имеются дополнительные сведения на Rational Unified Process Resource Center, в которых описаны алгоритмы для оценки размера исходя из прецедентов. Когда вы будете использовать эти страницы, вы должны учитывать расхождения, возможные между уровнем абстракции прецедентов в различных организациях и проектах. Один раз выбрав уровень детализации прецедентов, важно его сохранить как стандапртный стиль описания Прецедентов, иначе оценка размера может быть очень ошибочной.

Оценка общей трудоемкости и цены

Потребности в кадрах и расписание проекта может быть вычислено исходя из размеров продукта при использовании научных моделей. Сегодня господствуют две заметные модели: COnstructive COst MOdel (COCOMO), разработанная Barry Boehm, и Larry Putnam's Putnam Methodology. Обе модели проверены в производстве. Сайт COCOMO: COCOMOII web site.

Отдельно стоит вопрос о производительности проектной команды. Это значение определяет в целом трудоемкость проекта. Полное проектное расписание нелинейно зависит от полной трудоемкости. Модели математически сложны и наилучшее решение - использовать программные средства для вычислений.

Привязка ограничений и приоритетов

В разработке всегда имеются некоторые ограничения (например, продукт должен быть выпущен к определенной дате, цена должна быть ниже $850,000) или приоритеты (например, продукт нужен как можно скорее). Исходя из фиксированности размера программы, можно запланировать требуемые ресурсы. Здесь опять видна нелинейная зависимость, которая описывается специфическими математическими моделями, Здесь также лучше использовать программное обеспечение для расчетов.

Выбор оптимального расписания, трудозатрат и оценка стоимости

Теперь вы имеете широкий круг сценариев проекта, выберите один из них, наиболее удовлетворяющий нуждам проекта. Так вы создадите начальную картину общей длительности проекта и покажете неообходимые ресурсы и бюджет.

Задать milestone'ы фаз проекта

Цель
  • Задать точки, в которых будет проводиться формальное оценивание проекта.
  • Назначить ожидаемые трудозатраты и стоимости каждой фазы.

Общий план проекта вначале задает даты основных milestone'ов . Эта часть Плана служит как общий путь проекта и создается на фазе Начало.

Чтобы запланировать фазы проекта в начальном цикле разработки, вы можете высказать гипотезы, основываясь на:

  • Опыте с подобными проектами в той же предметной области.
  • Степени новизны проекта.
  • Специфичных ограничениях типа время отклика, распределение, безопасность.
  • Зрелость организации.

Используя оценки, основанные на собственном опыте в других подобных проектах, вы создаете начальный проектный бюджет распределяя оцениваемые усилия и стоимость по фазам проекта.

Задать цели, которые будут достигнуты на milestone'ах

Цель
  • Задать критерии, по которым будет оцениваться фаза.

Каждый milestone сфокусирован на определенной цели; каждый представляет хорошо-заданную точку перехода на следующую фазу.

Каждый milestone представляет собой критическую точку, на которой решается, идти дальше или дорабатывать что есть.

Задать число, длительность и цели итераций по фазам.

Цель
  • Определить, как много итераций будет на каждой фазе.
  • Определить связанные работы по итерациям.
  • Определить цели каждой итерации.

На каждой итерации выпускается некий артефакт, выпуск, который может использоваться для оценки прогресса и качества. Так как каждая итерация сфокусирована на чем-то своем, функциональность и полнота итерационных результатов различается. Цели итерации должны быть сформулированы достаточно четко, чтобы в конце итерации определить, достигнуты ли они. На ранних итерациях цели обычно выражаются в виде уменьшить риск; в поздних итерациях цели выражаются в метриках функциональной завершенности и качества.

Уточнить даты и объемы проекта по milestone'ам

Цель
  • Уточнить смету по информации, доступной в конце фазы Начало

К концу фазы Начало, фазы могут быть спланированы более точно, если принять в расчет:

  • Число прецедентов.
  • Сложность уже продуманных прецедентов.
  • Риски, технические и деловые.
  • Функциональные точки или метрики прецедентов.
  • Результаты прототипирования.

Этот грубый план обновляется в течении фазы Уточнение. Он служит базисом для построения плана проекта.

Определить требования к ресурсам.

Цель
  • Определить число и типы ресурсов, необходимых на кажой фазе/итерации.

Основываясь на ваших предположениях о трудоемкости и распределении работ во времени, вы задаете требуемые ресурсы. Для каждой фазы/итерации определите какие нужны сотрудники (по ролям), и какое их количество.

Разработать план завершения проекта

Цель
  • Разработать план завершения проекта.

План завершения проекта включен как раздел в Общий план проекта. Завершение проекта - это совокупность активностей, необходимых для завершения проекта, показывающих, что любые метрики собраны, а какой-то опыт зафиксирован для будущего использования.

Завершение проекта наступает, когда выполнены условия:

  • Завершены все намеченные работы, все созданные артефакты помещены под конфигурационныйы контроль.
  • Приемочное тестирование завершено и продукт официально принят Группой Качества и специалистами, проводившими внешнее тестирование
  • Продукт официально передан на тиражирование

Задать завершающие активности

Сначала перечислите действия, которые нужно провести на завершении проекта. Обычно это:

  • Встреча по окончанию проекта
  • Разработка отчета по окончанию проекта
  • Окончание персональных экспертиз проекта
  • Архивирование проектных артефактов
  • Перераспределение сотрудников, занятых в проекте
  • Обновление базы метрик организации метриками проекта для будущего использования.

Определить участников завершающих активностей.

Затем определите участников работ по завершению проекта.

Создайте план завершения проекта

Задайте расписание для завершающих активностей. Обычно, оно создается в Общем плане проекта в конце проекта.
