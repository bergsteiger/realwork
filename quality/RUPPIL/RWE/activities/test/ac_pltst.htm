Определить требования к тестированию

Цель
  • Определить предмет тестирования и рамки тестирования.

Требования к тестированию определяют что тестировать, рамки тестирования и цели. Также они используются для определения тестового покрытия.

Требования, определенные для тестирования должны быть проверяемыми. Они должны иметь ясный и измеримый результат. Требования, которые нельзя проверить, не могут являтся требованиями для тестирования.

В ОИТ тестовыми требованиями для функционального, нагрузочного тестирования и тестирования на устойчивость являются требования, обозначенные в базе требований RequisitePro как

  • Основной поток и подпотоки
  • Специальные требования

Оценить риски

Цель
  • Максимизировать отдачу тестовых работ.
  • Установить наилучшую последовательность тестов.

Данный шаг направлен на выявление наиболее часто используемых пользователями функций. Чем чаще функция используется - тем тщательнее ее нужно проверять. В нагрузочном и стрессовом тестировании это является одним из основных источников информации для проектирования тестов.

Один из методов - наблюдение за пользователями, как они взаимодействуют с системой, возможно, с использованием программных мониторов-рекордеров для автоматической записи действий пользователя в системе (с последующим анализом).

Разработать стратегию тестирования

Цель
  • Определить и связать технику и инструменты тестирования
  • Определить и связать методы оценки качества продукта и полноту тестирования

Целью выработки тестовой стратегии является оповещение всех как будет проводится тестирование и что будет считаться его успешным окончанием.

Разработка тестовой стратегии включает:

  • Определить и описать подходы к тестированию
  • Определение критериев тестирования
  • Определение всех специальных ограничений

подходы к тестированию описывают как тестирование будет реализовано. Что тестируется, какие действия будут проводиться, как будут проверятться результаты.
Например,

  • Каждый прецедент преобразуется в тест-установку, для него будут определены корректные и некорректные входные данные
  • Для каждого потока прецедента будет создана тестовая процедура.
  • 1500 вирутальных пользователей будут выполнять функции A, B и C при разных входных данных.

Критерии тестирования показывают значения, используемые для определения момента, когда тестирование завершено, и качество тестируемого приложения. Тестовые критерии должны определить:

  • предмет тестирования
  • как будут проводиться измерения
  • критерии оценки измерений

Определить ресурсы

Цель
  • Определить необходимые для тестирования ресурсы (люди - навыки, квалификация, опыт; машины, инструментарий).

Для проведения тестирования разных видов может быть нужны специальные навыки или ресурсы. Поэтому вопросы выделения ресурсов на проведение функционального, компонентного, устойчивости тестирований должны рассматриваться отдельно

Определите кто будет:

  • Управлять и планировать тестирование
  • Проектировать тесты и данные
  • Реализовывать тесты и данные
  • Запускать тесты и снимать результаты

Не-людские ресурсы: Операционные системы и все виды программного обеспечения должны быть рассмотрены. Тесты как правило необходимо повторять на всех возможных конфигурациях ОС и прикладного ПО.

Отдельный момент при проведении функционального тестирования - антивирусные и прочие работающие в фоновом режиме программы.

При проведении нагрузочного тестирования и тестирования на устойчивость отдельные аппаратные ресурсы должны быть отданы в эксклюзивный доступ.

Создать расписание

Цель
  • Определить и связать работы, расписание и milestone'ы

Создание расписания включает:

  • Оценку трудоемкости
  • Генерирование тестового расписания

Оценка трудоемкости проводится Менеджером по Качеству ОИТ. В частности, рассматривается:

  • Продуктивность и квалификация человеческих ресурсов
  • Объемы приложения (количество окон, компонент, сложность БД, процент вторичного использования компонент и тестовых скриптов
  • Приемлемая глубина тестового покрытия
  • Объемы компонентного тестирования

Тестовые усилия должны включать время для проведения регрессионных тестов.

Тестовое расписание составляется аналогично Планам итерации, за исключением, самих работ.

Сгенерировать план тестирования

Цель
  • Собрать воедино всю информацию по планированию тестирования.

Проведите следующее:

  • Проведите обзор существующих материалов
  • Определите результаты тестирования
  • Создайте план тестирования

Сначала проведите экспертизу всей имеющейся информации, которую нужно включить в план тестирования.

Определите результаты тестирования и то, как они будут создаваться, поддерживаться, становиться доступными для остальных членов команды. Результаты включают:

  • Тестовые установки
  • Тестовые скрипты
  • Запросы на изменение
  • Итоговые отчеты о тестировании
