Активность: Смоделировать пользовательский интерфейс

Activity: Model the User-Interface

Цель
Purpose
  • Создать модель пользовательского интерфейса, которая обладает необходимыми удобством и практичностью.
  • To build a model of the user interface that supports the reasoning about, and the enhancement of, its usability.
Шаги
Steps

Для каждого прецедента, основываясь на перспективе практичности в текущей итерации, проведите шаги:

For each use case prioritized to be considered from a usability perspective in the current iteration, take the following steps:

  • Описать характеристики связанных актеров
  • Создать раскадровку прецедента
  • Описать поток событий - Раскадровку
  • Установить требования практичности на раскадровке прецедента
  • Выделить граничные классы, участвующие в раскадровке прецедента
  • Описать взаимодействия между граничными классами и актерами
  • Дополнить диаграммы раскадровки прецедента
  • Сослаться от раскадровки прецедента к прототипу пользовательского интерфейса

  • Describe the Characteristics of Related Actors
  • Create a Use-Case Storyboard
  • Describe the Flow of Events - Storyboard
  • Capture Usability Requirements on the Use-Case Storyboard
  • Find Boundary Classes Needed by the Use-Case Storyboard
  • Describe Interactions Between Boundary Objects and Actors
  • Complement the Diagrams of the Use-Case Storyboard
  • Refer to the User-Interface Prototype from the Use-Case Storyboard

Для выявленных граничных классов проведите следующие шаги:

For the identified boundary classes, take the following steps:

  • Описать ответственности граничных классов
  • Описать атрибуты граничных классов
  • Описать связи между граничными классами
  • Описать требования практичности на граничных классах
  • На глобальных диаграммах классов показать граничные классы
  • Оценить результаты

  • Describe Responsibilities of Boundary Classes
  • Describe Attributes of Boundary Classes
  • Describe Relationships between Boundary Classes
  • Describe Usability Requirements on Boundary Classes
  • Present the Boundary Classes in Global Class Diagrams
  • Evaluate Your Results

Примечание: эти шаги представлены в логичном порядке, но вы можете его изменить или проводить несколько одновременно. Также, некоторые шаги могут быть пропущены, в зависимости от сложности конкретного пользовательского интерфейса.

Note: these steps are presented in a logical order, but you may have to alternate between them, or perform some of them in parallel. Also, some steps are optional depending on the complexity of the specific user interface under consideration.

Описать характеристики связанных актеров

Describe the Characteristics of Related Actors

Описание характеристик актеров-людей связано с прецедентом. Сосредоточьтесь на описании первичного актера прецедента, так как он вовлечен в большую часть взаимодействий. Эта информация важна для шагов, описанных ниже.

Describe the characteristics of the (human) actors related to the use case. Focus on describing the primary actor of the use case since the major part of the interaction involves this actor. This information is important for the subsequent steps below.


Обратитесь к Руководство: Актер, Характеристики для деталей.

Refer to Guidelines: Actor, Characteristics for details.

Создать раскадровку прецедента

Create a Use-Case Storyboard

Начните с создания раскадровки прецедента и установите зависимость между двумя. dependency between the two. Раскадровка прецедента обладает различными свойствами, такими как текстовые поля и диаграммы, которые описывают аспекты удобства прецедента.

Start by creating a use-case storyboard for the use case, and assign a trace dependency between the two. The use-case storyboard has various properties, such as text fields and diagrams, that describe the usability aspects of the use case.

Раскадровка прецедента в модели анализа может быть связана с прецедентом Модели прецедентов.


A use-case storyboard in the analysis model can be traced to a use case in the use-case model.


См. Руководство: Раскадровка прецедента для дополнений.

Refer to Guidelines: Use-Case Storyboard for details.

Описать поток событий - Раскадровку

Describe the Flow of Events - Storyboard

Пошаговое описание каждого прецедента, которое является входящим артефактов в эту активность, должно быть уточнено и сосредоточено на вопросах удобства; это усовершенствование фиксируется в свойстве Поток Событий - Раскадровка раскадровки прецедента.

The step-by-step description of each use case that is input to this activity need to be refined and focused on usability issues; this refinement is captured in the Flow of Events - Storyboard property of the use-case storyboard.


Начиная с создания начального очертания Потока Событий - Раскадровки, используя краткие описания действий, получите понимание прецедента.

Start by creating an initial outline of the Flow of Events - Storyboard using brief action statements, to acquire an understanding of the use case.


Затем дополните Потока Событий - Раскадровку с желаемым управлением, средними значениями и объемами объектов, и средними действиями,используемыми в различных частям потока событий.

Then complement the Flow of Events - Storyboard with desired guidance, average values and volumes of objects, and average action usages for the different parts of the flow of events.


См. Руководство: Раскадровка прецедента, Описание потока событий для дополнительной информации.

Refer to Guidelines: Use-Case Storyboard, Describing the flow of Events for details.

Установить требования практичности на раскадровке прецедента

Установите требования практичности на раскадровке прецедента, определив каким должно быть удобство интерфейса.

Capture the usability requirements on the use-case storyboard, defining how high the usability of the user interface must be.


См. Руководство: Раскадровка прецедента, Фиксация требований практичности для доп. информации.

Выделить граничные классы, участвующие в раскадровке прецедента

Find Boundary Classes Needed by the Use-Case Storyboard

В этом шаге мы определяем центральные и примитивные граничные классы, обеспечивающие пользовательский интерфейс прецедента.

In this step we identify the central and primitive boundary classes needed to provide the user interface of the use case.

Определить центральные граничные классы

Identify Central Boundary Classes

Основное окно, в котором работает пользователь, представляет один ценральный граничный класс. Если уже такой класс существует, продумайте возможность его вторичного использования. Это связано с тем, что всегда следует стремиться минимизировать количество первичных окон, в которых работает пользователь. Однако следует достигнуть компромиса между перегруженностью окон и их количеством.

Identify one central boundary class representing the primary window with which the actor is interacting. If such a class already exists, consider reusing it. This is because you should strive to minimize the number of primary windows a particular actor needs to interact with. However, you will, of course, have to compromise; primary windows should not be too complex or cluttered.


Центральные граничные классы часто включают в себя примитивные граничные классы (см. ниже).

These central boundary classes are often aggregates of more primitive boundary classes (see below).

Определить примитивные граничные классы

Identify Primitive Boundary Classes

Детально исследуйте поток событий - раскадровку, и все описания прецедентов. Ищите информацию, интересующую пользователя и информацию, которую нужно показывать или которой нужно управлять посредством пользовательского интерфейса.

Study the flow of events - storyboard in detail, and any other use-case descriptions at hand. Look for information interesting to the user, and information that needs to be visible and managed in the user interface.


Рассмотрите все объекты бизнес-сущности или термины (особенно существительные), содержащиеся в Глоссарии. Также рассмотрите все существующие сущности. Это путь к пониманию того, какая информация используется системой, а поэтому интересна и пользователю.

Consider any business entity objects, or terms (especially nouns) in a glossary. Also consider entity objects if they exist. These are ways of gaining an understanding of what information is manipulated by the system and is thereby also interesting to the user.


Затем создайте один граничный класс для каждого логического объекта, которым через пользовательский интерфейс управляет актер. Когда это будете делать, всегда рассматривайте возможность вторичного использования уже выделенных граничных классов. Результатом этого шага будет являться подготовка к построению объектно-ориентированного интерфейса.

Then, create one boundary class for each logical object manipulated by the actor via the user interface. When doing this, always consider reusing already existing boundary classes. The result of this step will prepare for the building of an object-oriented user interface.

Принципы определения граничных классов

Outline the Identified Boundary Classes

Опишите явные вещи о граничных классах. Здесь зачастую возможно выделить большинство их атрибутов, агрегации, ассоциации и часть ответственностей. Помните, что центральные граничные классы часто являются совокупностями более примитивных граничных классов.

Describe obvious things about the boundary classes. At this stage, it is often possible to outline most of their attributes, aggregations, associations, and also some of their responsibilities. Recall that central boundary classes are often aggregations of more primitive boundary classes.


Каждый класс будет уточнен и детально описан в следующих шагах. В них указаны и примеры.

Each class will be refined and described in detail in the steps below; please refer to the steps below for examples.

Отношения граничных классов и раскадровки прецедента

Relate the Boundary Classes to the Use-Case Storyboard

Граничные классы, участвующиев раскадровке прецедента должны иметь связи с ней. Это производится путем фиксации классов на диаграмме классов, включая их взаимосвязи, прикрепленные к раскадровкам прецедентов.

The boundary classes participating in a use-case storyboard need to be related to it. This is done by capturing the classes in class diagrams, including their relationships, attached to the use-case storyboards.


См. Руководство: Раскадровка прецедента, Создание диаграмм граничных классов

Refer to Guidelines: Use-Case Storyboard, Creating Boundary Class Diagrams for details.

Описать взаимодействия между граничными классами и актерами

Describe Interactions Between Boundary Objects and Actors

Этот шаг может не проводиться, если последовательность взаимодействия (например, последовательность появления окон и пути навигации) необходимо показать на диаграммах взаимодействий.

This step is optional and is done if the interaction sequence (e.g., flow of windows and navigation paths) needs to be captured formally in an interaction diagram.


Пройдите через поток событий прецедента (более точно - через набор рассматриваемых прецедентов в текущей итерации), выявляя граничные объекты, предназначенные для реализации визуального поведения прецедента. Заметьте, что эти граничные объекты являются экземплярами граничных классов, найденных на предыдущем шаге (Выделить граничные классы, участвующие в раскадровке прецедента). Проиллюстрируйте взаимодействия между граничными объектами путем создания кооперативных диаграмм.

Step through the flow of events of the use case (or more precisely, the subset of the use case under consideration in the current iteration), identifying the boundary objects responsible for the visible behavior of the use case. Note that these boundary objects are instances of the boundary classes as found in the previous step (Find Boundary Classes Needed by the Use-Case Storyboard). Illustrate the interactions between the participating boundary objects by creating one or more collaboration diagrams.


Кооперативные диаграммы также должны показывать и взаимодействие между системой и ее актерами. Поток событий обычно начинается, когда один из актеров запрашивает что-либо у системы, вызывая прецедент.

The collaboration diagram should also show interactions between the system and its actors. The flow of events usually begins when one of the actors requests something from the system, since an actor always invokes the use case.


Проиллюстрируйте взаимодействие между объектами путем создания связей между ними; кратко поясните эти связи, укажите цель вызова объекта, то есть что хочется достичь во взаимодействии с вызываемым объектом. Illustrate the interactions between objects by creating links between them; annotate the links with a short description of the intent of the invoking object, i.e. what it wants to achieve in the interaction with the invoked object. Эти "цели" будут затем введены в ответственности на каждом связанном граничном классе, как описано в следующем шаге (Описать ответственности граничных классов).

Illustrate the interactions between objects by creating links between them; annotate the links with a short description of the intent of the invoking object, i.e. what it wants to achieve in the interaction with the invoked object. These "intents" will then be integrated into responsibilities on each corresponding boundary class, as described in the following step (Describe Responsibilities of Boundary Classes).

Refer to Guidelines: Use-Case Storyboard, Creating Boundary Object Interaction Diagrams for details.

Описать ответственности граничных классов

Describe Responsibilities of Boundary Classes

Подход к этому шагу тот же самый, что был в разделе "Описать ответственности" в Активности: Провести анализ прецедента. Разница лишь в том, что потребности извлекаются из раскадровок прецедентов, а не из реализаций.

The approach to follow in this step is similar to the one presented in the section "Describe Responsibilities" as described in the Activity: Use-Case Analysis. The only difference is that the demands are extracted from use-case storyboards instead of use-case realizations.


Заметьте, что ответственность - это текстовое описание связанного набора поведений, обеспечиваемых классом. Ответственности граничных классов могут рассматриваться как высокоуровневые описания операций, обеспечиваемых пользовательским интерфейсом.

Note that a responsibility is a textual description of a cohesive subset of the behavior provided by a class. Given this, the responsibilities of boundary classes can be considered as high-level descriptions of the operations provided by the user interface and its windows.


Примеры находятся в разделе "Примеры ответственностей граничных классов" в Инструкции: Граничный Класс.

For examples, refer to section "Examples of Boundary Class Responsibilities" in Guideline: Boundary Class.

Describe Attributes of Boundary Classes

Подход к этому шагу тот же самый, что был в разделе "Описать атрибуты и ассоциации" в Активности: Провести анализ прецедента.

The approach to follow in this step is similar to the one presented in the section "Describe Attributes and Associations" as described in the Activity: Use-Case Analysis.


Заметьте, что атрибуты и их типы должны быть концептуальными и служить высокоуровневыми описаниями свойств объектов (таких как окна) в пользовательском интерфейсе. Также эти атрибуты могут стать классами сами по себе, когда будет проектироваться и реализовываться пользовательский интерфейс.

Note that the attributes and their types should be conceptual, and serve as high-level descriptions of the properties of objects (such as windows) in the user interface. Also, these attributes may become classes themselves when the user interfaces is designed and implemented.

However, in some cases it is of value to present the boundary classes in "global" class diagrams that are not local to any use-case storyboard; for example, when used as input to the user-interface prototyping activity. For example, such class diagrams can include the following:

  • Самые важные граничные классы и их взаимосвязи. Диаграммы этого типа могут служить итогом модели и сильно помогать например, при ее обзоре или установлении приоритетов текущей итерации.
  • Граничные классы, вместе задающие пользовательских интерфейс.
  • Важные, или глубокие, иерархии агрегаций граничных классов.
  • Ассоциации между граничными классами в различных иерархиях агрегации.
  • Иерархии наследования граничных классов.

  • The most important boundary classes and their relationships. Diagrams of this type can function as a summary of the model, and can be of great help in, for example, a review of the model, or in a prioritization of which parts of the model will be focused upon in the current iteration.
  • Boundary classes that together constitute a specific user interface.
  • Important, or deep, aggregation hierarchies of boundary classes.
  • Associations between boundary classes in different aggregation hierarchies.
  • Generalization hierarchies of boundary classes.
