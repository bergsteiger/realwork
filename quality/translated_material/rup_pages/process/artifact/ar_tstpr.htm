Тест-Процедура

 

Test Procedure

 Тест процедура - набор детальных инструкций для установки, выполнения, и оценки результатов для данной тест-установки (или набора тест-установок).

A test procedure is a set of detailed instructions for the set-up, execution, and evaluation of results for a given test case (or set of test cases).

Цель

Purpose  

Цель тест-процедуры состоит в том, чтобы выявить и представить информацию, необходимую тестеру соответственно для  последовательной установки, реализации и/или выполнения тест-установки(ок).

The purpose of the test procedure is to identify and communicate the information necessary for the tester to set-up and implement and / or execute the test case(s) appropriately and consistently.

Следующие люди используют тест-процедуру:

The following people use the test procedure:

  • Разработчик Теста - для выявления тест-процедур, основанных на тест-установках, и структурирования тест-процедур, для достижения высокого уровня повторного использования и способности системы к восстановлению функций.

 

  • The Test Designer to identify test procedures based on the test cases and structure test procedures to obtain a high level of reuse and resilience.

Краткое описание

Brief Outline

1. Шаги / Действия

Steps / Actions

Ряд сжатых утверждений, отображающих шаги или действия, предпринимаемые актером при выполнении тест-установок (для данного прецедента).

A series of succinct statements indicating the steps or actions taken by the actor when executing a test case (for a given use case).

2. Входные Величины / Тест Установка

Input Values / Test Case

Реальные величины вводятся актером в каждом шаге/действии в тест-процедуре. Это также может использоваться как ссылка на тест-установку.

The actual values input by the actor at each step / action in the test procedure. This may also be used as a reference to a test case.

3. Ожидаемый Результат(ы)

Expected Result(s)

Ожидаемый ответ от приложения на данный шаг/действие. Это может быть установлено как режим машины/приложения (например как доступные открытие окна или кнопка на экране дисплея), или данные (такие как результат вычисления или восстановленная запись).

The expected response from the application for a given step / action. This may be stated as a machine / application state (such as a window appears or a pushbutton is enabled), or data (such as the result of a calculation or retrieved record).

4. Метод(ы) Проверки

Verification Method(s)

Метод используемый  для сравнения ожидаемых и фактических результатов. Метод проверки может быть простым визуальным наблюдением (появляющееся окно) или более сложной комбинацией визуальных и не-визуальных свойств (появляющееся окно в определенном режиме, местоположении и в определенный период времени).

The method used to compare the expected and actual results. The verification method may be a simple visual observation (window appeared) or a more complex combination of visual and non-visual properties (window appeared in a given state at a given location within a given time).

Согласование по времени

Timing

Начальная Тест-Процедура(ы) создается в процессе проектирования теста в течение Разработки, как только определенные прецеденты или сценарии выявляются, изменяются и/или добавляются, на протяжении оставшейся части жизненного цикла в течение каждой итерации.

The initial Test Procedure(s) are created during the test design activity during Elaboration, as soon as there are some use cases or scenarios defined and are modified and / or added throughout the remainder of the lifecycle during each iteration.

Ответственность

Responsibility

Разработчик Теста ответствен за целостность тест-процедуры и  гарантирует что:

  • Тест-процедуры точно отражают содержание итерации.
  • Тест-процедуры имеют соответствующее содержание и детализируют что необходимо для установки, реализации, выполнения, и оценки тест-установок.
  • Тест-процедуры отражают выполнение прецедента в согласовании с эмулируемым актером (с использованием той же самой навигации, методов интерфейса и средств).
  • The test procedures accurately reflect the contents of the iteration.
  • The test procedures contain the appropriate content and detail necessary to set-up, implement, execute, and evaluate the test cases.
  • The test procedures reflect the execution of the use case consistent with the actor being emulated (using the same navigation, interface methods and options).

Адаптация

Tailoring  

Тест-Процедуры (содержание и формат) могут нуждаться в модификации, для достижения требований внутренних или внешних стандартов, руководств, или контроля.

The Test Procedures (contents and format) may require modification to meet the needs of internal or external standards, guidelines, or oversight.
