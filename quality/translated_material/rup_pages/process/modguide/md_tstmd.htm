Тест-Модель

Test Model

 Тест- модель является представлением того, что и как будет тестироваться. Это вид проекта и моделей реализации, включающий собрание тест-установок, тест-процедур, и тест-сценариев, и их отношений.  

The test model is a representation of what will be tested and how it will be tested. It is a view of the design and implementation models and includes the collection of the test cases, test procedures, and test scripts, and their relationship.

Объяснение

Explanation

Тест-модель - это модель того, что должно быть тестировано в цели теста (будь это система или часть системы). Это вид проекта и моделей реализации, который включает дополнительную систему обозначений , специфичную для теста, включая тест-установки, тест-процедуры и тест-сценарии. Роль тест-модели состоит в том, чтобы передать, что и как должно быть проверено.

A test model is a model of what is to be tested in a target-of-test (be it the system or a part of the system). It is a view of the design and implementation models that include additional notation specific to test, including the test cases, test procedures, and test scripts. The role of the test model is to communicate what is to be tested and how to test it.

Использование

Use

Для каждой итерации в разработке жизненного цикла Вы развиваете новую версию тест-модели. Новая версия должна содержать старые тесты (возвратные тесты) и новые тесты, которые учитывают новые функциональные возможности.

For each iteration in the development lifecycle, you develop a new version of the test model. The new version should contain old tests (as regression tests), and new tests that take new functionality into account

На высшем уровне детализации тест-модели Вы увидите проект и/или модель реализации и примечания, отражающие тест-установки. Тест-установки определяют набор исходных данных, условий выполнения и ожидаемых результатов. Большинство тест-установок получаются из сценариев прецедента или реализаций прецедента и соответствует выполнению прецедента, поскольку это пересекается с целью теста. Если целью теста является целая система, то тест-установка будет соответствовать полному выполнению прецедента. Если цель теста является подсистемой, то тест-установка будет соответствовать части прецедента, которая пересекается с подсистемой.

At the least detailed level of the test model, you will see the design and / or implementation model and notations reflecting the test cases. Test cases define a collection of inputs, execution conditions and expected results. The majority of the test cases are derived from use-case scenarios, or use-case realizations, and correspond to an execution of a use case as it traverses the test target. If the target-of-test is the whole system, the test case will correspond to a complete execution of a use case. If the target-of-test is a subsystem, the test case will correspond to the part of the use case that traverses the subsystem.

Пример системной тест-установки для БАНКОМАТА должен был бы подтвердить, что клиент может снять 100$, если карта действительна и на определенном счете достаточно средств. Другая тест-установка должна была бы проверить, что программное обеспечение Банкомата ведет себя соответствующим образом, когда клиент вводит неправильный PIN код для другой действительной карты.

An example of a system test case for an ATM machine would be to verify that a customer can withdraw $100, provided the card is valid and enough funds are in the specified account. Another test case would be to verify that the ATM software behaves as expected when a customer enters the wrong PIN for an otherwise valid card.

Кроме того, в более ранних итерациях, может не быть каких-либо полных описаний прецедентов, только сценарии. Поэтому в первой итерации теста реализована только тест-установка для того, чтобы проверить возможность снимать наличные деньги, но не возможность обрабатывать неправильный PIN код.

In earlier iterations, there might not be any complete use-case descriptions yet, only scenarios. Therefore, in the first iteration of test, only a test case to verify the ability to withdraw cash is implemented, but not the ability to handle the wrong PIN code.

Наиболее детальная из тест-моделей отобразит отношения между тест-установками, тест-процедурами и тест-сценариями, которые создаются, чтобы выполнять тесты.

The most detailed of test models will depict the relationships between the test cases, test procedures, and the test scripts that are generated to execute the tests.

Не все требования в системе проявляются в прецедентах. Этот тип требования относится к не-функциональным или техническим требованиям. Примерами являются  требования к загрузке и требования к платформе приложения.

Not all requirements on a system are manifested in use cases. This type of requirement is referred to as non-functional or technical requirements. Examples are load requirements and application platform requirements.



Тест-установка получается из и прослеживается до прецедента или сценария, реализации прецедента или требования.

 A test case is derived from and traceable to a use case or scenario, a use-case realization, or a requirement.

В организациях, где возможность отслеживания от первичных требований системы до деталей реализации является важной (или востребованной), должна поддерживаться связь отслеживаемости между тест-установкой и соответствующим требованием(ми) системы.

In organizations where traceability from original system requirements to the details of the implementation is important (or required), a traceability link from a test case to its corresponding system requirement(s) should be maintained.

Обратите внимание: если Вы не имеете инструментария, который сможет эффективно помочь Вам поддержать эти связи отслеживаемости, то управление может стать очень дорогим. В этом случае, знание о том, из какого прецедента, сценария прецедента или реализации прецедента получается тест-установка может быть достаточно.

Note: if you do not have tools that efficiently help you maintain these traceability links, it may become very costly to administer. In that case knowing what use case, use-case scenario, or use-case realization a test case is derived from may be sufficient.

На самом высоком уровне, ваши тест-установки должны быть сгруппированы согласно тому, что является целью теста: система, подсистема, или компонент. На следующем уровне, Вы могли бы выбирать согласно тому, какому прецеденту или требованию, из которых получены тест-установки, производить группировку.

At the highest level, your test cases should be grouped according to what the target for test is: system, subsystem, or component. At the next level, you might choose to group according to what use case or requirement the test cases are derived from.

В простой реализации, тест-установка реализуется как одна тест-процедура. Тест-процедура физически проявляет себя как текстовая инструкция относительно того, как исполнить тест-установку (ручной тест), или как тест-сценарий, который будет выполнен в контексте инструментария автоматизации теста. В более сложной реализации, тест-установка может соответствовать одной и более тест-процедурам, также как одна тест-процедура может являться частью одной и более реализуемых тест-установок. Точно так же тест-процедуры могут быть реализованы одним и более тест-сценариями.

In a simple implementation, a test case is implemented as one test procedure. The test procedure physically manifests itself as a textual instruction on how to perform the test case (manual test), or as a test script to be executed within the context of a test automation tool. In a more complex implementation, a test case may correspond to one-to-many test procedures, as well as one test procedure being part of implementing one-to-many test cases. Similarly, the test procedures may be implemented by one-to-many test scripts.

See Artifacts: "Test Case", "Test Procedure", or "Test Script" for more information.

 

Отношения между тесит-установками, тест-процедурами, и тест-сценариями.

 The relationship between test cases, test procedures, and test scripts.

Тест-установка реализуется одной и более тест-процедурой.

A test case is implemented by one-to-many test procedures.

Тест-процедура может реализовывать (целую или части) одной и более тест-установки.

A test procedure may implement (the whole or parts of) one-to-many test cases.

Когда должно быть выполнено ручное тестирование, имеет место непосредственное отношение с тест-процедурами.

When manual testing is to be executed, there is a one-to-one relationship with the test procedures.

Когда тестирование автоматизировано (реализованы тест-сценарии), тест-процедура может быть реализована многими тест-сценариями или тест-сценарий может реализовывать много тест-процедур (или частей тест-процедур).

When testing is automated (test scripts implemented), a test procedure may be implemented by many test scripts, or a test script may implement many test procedures (or parts of test procedures).
