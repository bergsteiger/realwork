План Тестирования

 


Test Plan

План тестирования содержит информацию о цели и задачах тестирования в проекте. Также план тестирования определяет стратегии, которые нужно использовать для реализации и выполнения тестирования, и необходимые ресурсы.

The test plan contains information about the purpose and goals of testing within the project. Additionally, the test plan identifies the strategies to be used to implement and execute testing, and the resources needed.

Краткий обзор

Overview

Цель тест-плана состоит в том, чтобы свести воедино цели тестирования. Очень важно, чтобы этот документ был создан как можно раньше. Создайте его в первых итерациях стадии Разработки, и это не будет очень рано. Желательно, чтобы разработка тест-плана шла итеративно, с добавлением новых разделов по мере доступности информации.

The purpose of the test plan is to communicate the intent of the testing activities. It is critical that this document be created as early as possible. Generating this artifact early in one of the first iterations of the Elaboration phase would not be too early. It may be desirable to develop the test plan iteratively, adding sections as the information is available.

Тщательно должно проводиться установление четких связей между возможностями тестирования, стратегиями теста. Эта информация определяет цель и границы объема работ теста, что и как будет тестировано, и какие ресурсы для этого необходимы. Четкое формулирование этой информации ускорит обзор, обратную связь, и утверждение объема работ теста.

Care should be taken in clearly communicating the scope of testing, the requirements for test, the test strategies, and the resource needs. This information identifies the purpose and boundaries of the test effort, what will be tested, how it will be tested, and what resources are needed for testing. Stating this information clearly, will expedite the review, feedback, and approval of the test effort.

В начале проекта должен быть создан план тестирования, определяющий  все предназначенное для проекта тестирование, и который называется "Основной План Тестирования". По мере планирования каждой итерации, создается более точный "План Тестирования Итерации" (или несколько планов тестирования, упорядоченных по типу теста), содержащий только данные, имеющие отношение к итерации  (требования для теста, стратегии теста, ресурсы, и т.д.). Дополнительно эта информация может быть включена в План Итерации, при этом последний не станет слишком трудным для управления или использования.

  At the outset of the project, a test plan identifying the overall intended testing for the project should be created, called the "Master Test Plan." As each iteration is planned, a more precise "Iteration Test Plan" is created (or several test plans, organized by type of test), containing only the data (requirements for test, test strategies, resources, etc.) that pertain to the iteration. Alternately, this information may be included in the Iteration Plan, if it does not make the iteration plan too difficult to manage or use. 

Ниже приведены некоторые руководства для лучшего определения и передачи требований к тесту, рисков, приоритетов и стратегий теста.

Below are some guidelines to better identify and communicate the requirements for test, test risks and priorities, and test strategies.

Определение Требований к Тестам

Identifying Requirements for Test

Требования к тесту определяют то, что будет тестировано. Они являются особой целью теста. Имеется несколько общих правил, которые необходимо применять при выявлении требований для теста:

Requirements for test identify what will be tested. They are the specific target of a test. There are a few general rules to apply when deriving requirements for test:

  • Требование к тесту должно быть доступным для наблюдения, с измеряемым характером изменений. Если требование к тесту не поддается наблюдению или не может быть измеренным, то оно и не может быть оценено для определения было ли требование удовлетворено.
  • Между каждым прецедентом или дополнительным требованием системы и требованием к тесту нет соотношения один к одному. Прецеденты часто будут иметь более одного требования к тесту, в то время как некоторые дополнительные требования будут порождать одно или более требований к тесту, а некоторые не произведут ни одного (такие как требования маркетинга или упаковки).
  • The requirement for test must be an observable, measurable behavior. If the requirement for test cannot be observed or measured, it can't can be assessed to determine if the requirement has been satisfied.
  • There is not a one-to-one relationship between each use case or supplemental requirement of a system and a requirement for test. Use cases will often have more than one requirement for test, while some supplemental requirements will derive one or more requirements for test and others will derive none (such as marketing or packaging requirements).

Требования к тесту могут быть получены из многих источников, включая прецеденты, модели прецедентов, дополнительные спецификации, требования проекта, деловые случаи, интервью с конечными пользователями и документ программной архитектуры. Все они должны быть рассмотрены, чтобы собрать информацию,  используемую для определения требований к тесту.

The requirements for test may be derived from many sources, including use cases, use-case models, supplemental specifications, design requirements, business cases, interviews with end-users, and the software architecture document. All of these should be reviewed to gather information that is used to identify the requirements for test.

Требования к Функциональным Тестам

Requirements for Functional Tests

Функциональные требования к тесту, как подразумевают их названия, выявляются из описаний функциональных изменений цели тестов. Как минимум  каждый прецедент должен порождать хотя бы одно требование к тесту. Более подробный список требований к тесту включил бы по крайней мере одно требование к тесту для каждого потока событий прецедента.

Functional requirements for test, as their name implies, are derived from descriptions of the target-of-test's functional behaviors. At a minimum, each use case should derive at least one requirement for test. A more detailed list of requirements for test would include at least one requirement for test for each use case flow of events.

Требования к Тестам Производительности

Requirements for Performance Tests

Требования эффективности к тесту выводятся из изменений производительности, определенных целью тестов . Как правило, производительность определяется как показатель времени ответа и/или использования ресурса, измеряемое при различных условиях, включающих

Performance requirements for test are derived from the target-of-test's specified performance behaviors. Typically, performance is stated as a measure of response time and/or resource usage, as measured under various conditions, including

  • Различные объемы работ и/или состояния системы
  • Различные прецеденты
  • Различные конфигурации
  • different workloads and/or system conditions
  • different use cases
  • different configurations

Требования к производительности описаны в Дополнительных Спецификациях. Рассмотрите эти материалы, придавая особое внимание утверждениям, включающим следующее:

Requirements for performance are described in the Supplementary Specifications. Review these materials, paying especial attention to statements that include the following:

  • Установки времени, такого как время ответа или согласования по времени профилей
  • Утверждения, указывающие, что определенное количество событий или прецедентов должно произойти в определенный период времени
  • Утверждения, сравнивающие поведение одного элемента по отношению к другому
  • Утверждения, сравнивающие поведение приложения в одной конфигурации к таковому в другой конфигурации
  • Эксплуатационная надежность (означает время до отказа или MTTF) в течение определенного периода времени
  • Конфигурации или ограничения
  • statements of time, such as response time or timing profiles
  • statements indicating that a number of events or use cases must occur in a stated period of time
  • statements comparing the behavior of one item to another
  • statements comparing the application behavior on one configuration to that of another
  • operational reliability (mean time to failure or MTTF) over a period of time
  • configurations or constraints

Вы должны получить по крайней мере одно требование к тесту для каждого утверждения в спецификации,  в которой информация отражается так, как было указано выше.

You should derive at least one requirement for test for each statement in the specification which reflects information such that listed above.

Требования к Тестам Надежности

Requirements for Reliability Tests  

Требования Надежности к тесту выводятся из нескольких источников, стандартно описанных в Дополнительных Спецификациях, Руководстве по Интерфейсу пользователя, Руководстве по Проекту, и Руководстве по Программированию.

Reliability requirements for test are derived several sources, typically described in Supplementary Specifications, User-Interface Guidelines, Design Guidelines, and Programming Guidelines.

Рассмотрите эти артефакты, и особое внимание уделите утверждениям, которые включают следующее:

Review these artifacts and pay especial attention to statements that include the following:

  • Надежность установок или резистентность к отказу, ошибки при выполнении (такие как нехватка памяти)
  • Установки, отображающие целостность и структуру кода (согласованность с языком и синтаксисом)
  • Установки касающиеся использования ресурса
  • statements reliability or resistance to failure, run-time errors (such as memory leaks)
  • statements indicating code integrity and structure (compliance to language and syntax)
  • statements regarding resource usage

Из каждого утверждения в артефактах, отражающих перечисленную выше информацию, должно быть получено по крайней мере одно требование к тесту.

At least one requirement for test should be derived from each statement in the artifacts that reflects information listed above.

Для успешного тестирования требуется, чтобы в тесте были уравновешены такие факторы, как ограничения ресурсов и риски. Для этого объем работ теста должен быть упорядочен по приоритетам так, чтобы наиболее важные, существенные, или наиболее подверженные рискам прецеденты или компоненты были тестированы раньше других. Чтобы произвести упорядочение по приоритетам нужно выполнить оценку риска и функциональный разрез, которые используются в качестве основания для установления приоритета теста.

Successful testing requires that the test effort successfully balance factors such as resource constraints and risks. To accomplish this, the test effort should be prioritized so that the most important, significant, or riskiest use cases or components are tested first. To prioritize the test effort, a risk assessment and operational profile are performed and used as the basis for establishing the test priority.

В следующих разделах описывается как определить приоритет теста.

The following sections describe how to determine test priority.

Оцените Риск, и Установите Приоритеты Теста

Assess Risk and Establish Test Priorities

Определение требований к тесту - это только часть определения того, что будет тестировано. Также должно быть выполнено упорядочение по приоритетам того, что будет тестировано и в каком порядке это будет произведено. Этот действие выполняется по нескольким причинам, включающим:

Identifying the requirements for test is only part of identifying what will be tested. Prioritizing what will be tested and in what order should also be performed. This step is done for several reasons, including:

  • Обеспечить, чтобы действия теста были сосредоточены на наиболее подходящих требованиях к тесту
  • Обеспечить, чтобы наиболее важные, существенные, или наиболее подверженные рискам требования к тесту были выявлены как можно раньше
  • Обеспечить, чтобы любые зависимости (последовательность, данные, и т.д.) были учтены в тестировании
  • ensure the test efforts are focused on the most appropriate requirements for test
  • ensure the most critical, significant, or riskiest requirements for test are addressed as early as possible
  • ensure that any dependencies (sequence, data, etc.) are accounted for in the testing

Оценка риска и установление приоритетов теста выполняются в три этапа:

There are three steps to assessing risk and establishing the test priorities:

  • Оцените Риск
  • Определите Функциональный разрез
  • Установите Приоритет Теста
  • Assess Risk
  • Determine Operational Profile
  • Establish Test Priority

Оцените Риск

Assess Risk

Установление приоритета для теста начинается с оценки риска. Прецеденты или компоненты, которые вызывают самый большой риск по причине отказа или имеют высокую вероятность отказа, должны быть проверенны одними из первых.

Establishing the priority for test begins with the assessment of risk. Use cases or components that pose the greatest risk due to failure or have a high probability of failure should be amongst the first use cases tested.

Начните с определения и описания показателей величины риска, которые будут использоваться, таких как:

Begin by identifying and describing the risk magnitude indicators that will be used, such as:

  • H - высокий риск, недопустимый. Серьезное внешнее воздействие. Компания будет нести большие финансовые потери, ответственность, или невосстанавливаемую потерю репутации.
  • M - средний риск, допустимый, но нежелательный. Минимальное внешнее воздействие, компания может страдать материально, но имеет место ограниченная ответственность или потеря репутации.
  • L - низкий риск, допустимый. Отсутствие или  небольшое внешнее воздействие, компания не имеет или имеет небольшие финансовые потери или ответственность. Репутация компании не затронута.
  • H - high risk, not tolerable. Severe external exposure. The company will suffer great financial losses, liability, or un-recoverable loss of reputation.
  • M - medium risk, tolerable, but not desirable. Minimal external exposure, the company may suffer financially, but there is limited liability or loss of reputation.
  • L - low risk, tolerable. Little or no external exposure, company has little or no financial loss or liability. Company's reputation unaffected.

После определения показателей величины риска, внесите в цель теста каждый прецедент или компонент. Для каждого прецедента или компонента в вашем списке, определите показатель величины риска, и обоснуйте (в кратком утверждении) выбранную Вами величину.

After identifying the risk magnitude indicators, list each use case or component in the target-of-test. For each use case or component in your list, identify a risk magnitude indicator, and justify (in a brief statement) the value you selected.

Имеются три перспективы, которые могут использоваться для оценки риска:

There are three perspectives that can be used for assessing risk:

  • Эффект - воздействие или последствие отказа указанного прецедента (требования, и т.д.) 
  • Причина - нежелательный результат, вызванный отказом прецедента
  • Вероятность - вероятность отказа прецедента.
  • Effect - the impact or consequence of a specified use case (requirement, etc.) failing
  • Cause - an undesirable outcome caused by the failure of a use case
  • Likelihood - the probability of a use case failing.

Выберите одну перспективу, определите показатель величины риска, и обоснуйте ваш выбор. Не обязательно определять показатель для каждой перспективы риска. Однако, предполагается, что если был выявлен низкий показатель риска, то надо попытаться оценить элемент по другой перспективе риска, чтобы подтвердить, что у элемента действительно низкий риск.

However, it suggested that, if a low indicator was identified, try evaluating the item from a different risk perspective to ensure the item is really a low risk.

Ниже более детально описана оценка риска с помощью этих трех перспектив.

Below are greater details on assessing risk by these three perspectives.

Эффект

Effect

Чтобы оценить риск по Эффекту, определите условие, событие, или действие, и попробуйте определить его воздействие. Задайте вопрос:

To assess risk by Effect, identify a condition, event, or action and try to determine its impact. Ask the question:

" Что случилось бы если ___________? "

"What would happen if ___________?"

Например:

For example:

  • " Что случилось бы, если при инсталляции нового программного обеспечения, системе бы не хватило места диске? "
  • " Что случилось бы, если связь с Интернетом была бы потеряна в процессе запроса сделки (трансакции) ? "
  • " Что случилось бы, если связь Интернета была бы потеряна при  выполнении сделки закупки? "
  • " Что случиться, если пользователь введет неожидаемое значение? "

 

  • "What would happen if while installing the new software, the system runs out of disk space?"
  • "What would happen if the Internet connection is lost during an inquiry transaction?"
  • "What would happen if the Internet connection was lost during a purchase transaction?"
  • "What would happen if the user enters an unexpected value?"

Причина

Cause

Оценка риска по Причине является противоположностью оценке по Эффекту. Начните с установки нежелательного случая или условия, и определите набор событий, которые могли бы позволить условию существовать. Задайте вопрос типа:

Assessing risk by Cause is the opposite of by Effect. Begin by stating an undesirable event or condition, and identify the set of events that could have permitted the condition to exist. Ask a question such as:

" Как могло ___________ случиться?

"How could ___________ happen?

Например:

For example:

  • " Как могли только некоторые из файлов, оказаться в системе и не была завершена регистрация? "
  • " Как сделка могла быть не отражена должным образом в центральной базе данных?
  • " Как может отчет  о процессе составления счетов отражать только некоторые из записей в базе данных, которые осуществляют желательные критерии? "
  • "How could only some of the files be on the system and not all the registry entries made?"
  • "How could a transaction not be reflected properly in the central database?
  • "How could the billing cycle statement reflect only some of records in the database that fulfill the desired criteria?"

Вероятность

Оценка риска по Вероятности должна определять вероятность того, что прецедент (или компонент, реализующий прецедент) потерпит неудачу. Вероятность обычно основана на внешних факторах таких как:

Assessing risk by Likelihood is to determine the probability that a use case (or component implementing a use case) will fail. The probability is usually based on an external factors such as:

  • Частота(ы) и/или плотность отказа 
  • Частота изменений
  • Сложность
  • Происхождение/Разработчик
  • Failure rate(s) and / or density
  • Rate of change
  • Complexity
  • Origination / Originator

Следует отметить, что при использовании этой перспективы риска, показатели величины риска связаны с вероятностью отказа, а не с эффектом или воздействием отказа на структуру, как это было при оценке риска по Эффекту и Причине.

It should be noted, that when using this risk perspective, the risk magnitude indicators are related to the probability of a failure, not the effect or impact the failure has on the organization as was used in assessing risk by Effect and Cause.

Failure discovery rate
and / or density

 Вероятность отказа увеличивается с увеличением частоты или плотности выявления отказа. Дефекты имеют тенденцию накапливаться, поэтому, как только частота или число дефектов (плотность)  увеличиваются в прецеденте или компоненте, то вероятность обнаружения другого дефекта также увеличивается. Также при оценке риска c использованием этого фактора должны рассматриваться частота выявления и плотность дефектов в предыдущих версиях, поскольку их высокие значения указывают на высокую вероятность дополнительных отказов.

The probability of a failure increases as the failure discovery rates or density increases. Defects tend to congregate, therefore, as the rate of discovery or the number of defects (density) increases in a use case or component, the probability of finding another defect also increases. Discovery rates and density from previous releases should also be considered when assessing risk using this factor, as previous high discovery rates or densities indicate a high probability of additional failures.
Частота изменений

 

 

 

Rate of change

 Вероятность отказа  увеличивается с увеличением частоты изменений в прецеденте или его компоненте. Поэтому, с ростом числа изменений также увеличивается и вероятность появления дефекта. Каждый раз при изменении  кода, имеется риск "введения" в него другого дефекта.

The probability of a failure increases as the rate of change to the use case or component increases. Therefore, as the number of changes increases, so too does the probability that a defect has been introduced. Every time a change is made to the code, there is the risk of "injecting" another defect it.
Сложность

 

Complexity

 Вероятность отказа увеличивается с увеличением показателя сложности прецедента или компонента.

The probability of a failure increases as the measure of complexity of the use case or component increases.
Происхождение/Разработчик

 

 

 

 

 

Origination / Originator

 Квалификация и опыт того, кем и где был произведен код, может увеличивать или уменьшать вероятность отказа.
Использование компонентов третьего лица обычно уменьшает вероятность отказа. Однако, это было бы верно только в том случае, если компонент третьего лица был сертифицирован (соответствовал вашим требованиям, на основании формального тестирования или опыта).
Вероятность отказа обычно уменьшается с увеличением знаний и навыков исполнителя. Однако, такие факторы как использование новых инструментов, технологий, или работа за нескольких исполнителей могут увеличивать вероятность отказа даже у самых лучших рабочих.

Knowledge and experience of where the code originated and by whom can increase or decrease the probability of a failure.
The use of third party components typically decreases the probability of failure. However, this is only true if the third party component has been certified (meets your requirements, either through formal test or experience).
The probability of failure typically decreases with the increased knowledge and skills of the implementer. However, such factors as the use of new tools, technologies, or acting as multiple workers may increase the probability of a failure even by the best workers.

Определите Функциональный разрез

Determine Operational Profile

Следующим шагом в оценке риска и установления приоритета теста является определение функционального разреза цели тестов.

The next step in assessing risk and establishing a test priority is to determine the target-of-test's operational profile.

Начните с определения и описания показателей величины функционального разреза, которые будут использоваться, таких как:

Begin by identifying and describing the operational profile magnitude indicators that will be used, such as:

  • H - используемый достаточно часто, много раз за период или многими актерами или прецедентами.
  • M - часто используемый , несколько раз за период или несколькими актерами или прецедентами.
  • L - нечасто используемый или используемый очень немногими актерами или прецедентами.
  • H - used quite frequently, many times per period or by many actors or use cases.
  • M - used frequently, several times per period or by several actors or use cases.
  • L - infrequently used or used by very few actors or use cases.

Показатель функционального разреза, который Вы выбираете, должен быть основан на частоте выполнения прецедента или компонента, включая:

The operational profile indicator you select should be based upon the frequency a use case or component is executed, including:

  • Число раз, когда ОДИН актер (или прецедент) выполняет прецедент (или компонент) за данный период времени, или
  • Число АКТЕРОВ (или прецедентов) которые выполняют прецедент (или компонент)
  • the number of times ONE actor (or use case) executes the use case (or component) in a given period of time, or
  • the number of ACTORS (or use cases) that execute the use case (or component)

Как правило, чем чаще используется прецедент или компонент, тем выше показатель функционального разреза.

Typically, the greater the number of times a use case or component is used, the higher the operational profile indicator.

После определения показателей величины функционального разреза, которые нужно использовать, внесите в цель испытания каждый прецедент или компонент. Определите показатель функционального разреза для каждого элемента в вашем списке и определите ваше обоснование для величины показателя. Для этой оценки может использоваться информация из документа анализа рабочей нагрузки (См. Артефакт: Документ Анализа Рабочей нагрузки).

After identifying the operational profile magnitude indicators to be used, list each use case or component in the target-of-test. Determine an operational profile indicator for each item in your list and a state your justification for the indicator value. Information from the workload analysis document (See Artifact: Workload Analysis Document) may be used for this assessment.

Установите Приоритет Теста

Establish Test Priority

Последним шагом в оценке риска и установлении приоритета теста является установление приоритета теста.

The last step in the assessing risk and establishing a test priority is to establish the test priority.

Начните с определения и описания показателей величины приоритета теста, которые будут использоваться, таких как:

  • H - обязан быть тестированным
  • M - должен быть тестирован только после того, как все H элементы будут тестированы
  • L - может быть тестирован, но только после того, как все H и М элементы будут тестированы
  • H - must be tested
  • M - should be tested, will test only after all H items are tested
  • L - might be tested, but not until all H and M items have been tested

После определения показателей величины приоритета теста, которые нужно использовать, внесите в цель теста каждый прецедент или компонент. Определите показатель приоритета теста для каждого элемента в вашем списке и сформулируйте ваше обоснование. Ниже приведены некоторые руководства для определения показателя приоритета теста.

After identifying the test priority magnitude indicators to be used, list each use case or component in the target-of-test. Determine a test priority indicator for each item in your list and a state your justification. Below are some guidelines for determining a test priority indicator.

Примите во внимание нижеследующее при определении показателей приоритета теста для каждого элемента:

Consider the following when determining the test priority indicators for each item:

  • Значение показателя величины риска определенное ранее
  • Значение величины функционального разреза определенное ранее
  • Описания актера (актеры опытны? терпимы к рабочему окружению?, и т.д.)
  • Договорные обязательства (будет ли цель теста приемлема, если прецедент или компонент не будут поставлены?)
  • the risk magnitude indicator value you identified earlier
  • the operational profile magnitude value you identified earlier
  • the actor descriptions (are the actors experienced?, tolerant of work-arounds?, etc.)
  • contractual obligations (will the target-of-test be acceptable if a use case or component is not delivered?)

Стратегии для установления приоритета теста включают:

Strategies for establishing a test priority include:

  • Используйте самое высоко оцененное значение фактора (риск, функциональный разрез, и т.д.) для каждого элемента в качестве общего приоритета.
  • Определите один оцененный фактор (риск, функциональный разрез, или другой) как являющийся наиболее существенным, и используйте значение этого фактора как приоритет.
  • Используйте комбинацию оцененных факторов, чтобы определить приоритет.
  • Использование весовой схемы, где отдельные факторы взвешены, и их значения и приоритет  вычислены на основании их веса
  • Use the highest assessed factor (risk, operational profile, etc.) value for each item as the overall priority.
  • Identify one assessed factor (risk, operational profile, other) as being the most significant and use that factor's value as the priority.
  • Use a combination of assessed factors to identify the priority.
  • Using a weighting schema where individual factors are weighed, and their values and priority calculated based upon the weight.

Стратегия Теста

Test Strategy

Стратегия Теста описывает общий подход и цели определенного объема работ теста.

The Test Strategy describes the general approach and objectives of a specific test effort.

Хорошая стратегия теста должна содержать следующее:

A good test strategy should contain the following:

  • Тип теста, который будет реализован, и его цель
  • Стадия, в которой тест будет реализован
  • Техника
  • Измерение и критерии используемые для оценки результатов теста и завершенности теста
  • Любые специальные соображения, которые затрагивают объем работ теста, описанный в стратегии теста
  • type of test to be implemented and its objective
  • stage in which test will be implemented
  • technique
  • measurement and criteria used to assess test results and test completion
  • any special considerations that affect the test effort described in the test strategy

Тип и Цель Теста 

Type of Test and Objective

Ясно сформулируйте тип и цели реализуемого теста. Это уменьшит путаницу и минимизирует недоразумения (тем более, что некоторые тесты могут выглядеть очень похожими). Ясно должна быть сформулирована цель - почему выполняется тест.

State clearly the type of test being implemented and the objective of the test. Explicitly stating this information reduces confusion and minimizes misunderstandings (especially since some tests may look very similar). The objective should state clearly why the test is being executed.

Примеры:

Examples:

"Функциональный Тест. Функциональный тест сосредоточен на выполнении последующих прецедентов из интерфейса пользователя, реализованных в цели теста."

"Functional Test. The functional test focuses on executing the following use cases implemented in the target-of-test, from the user interface."

"Тест Производительности. Тест производительности для  системы сосредоточен на измерении времени ответа для прецедентов 2, 4, и 8 - 10. Для этих тестов используется рабочая нагрузка одного актера, выполняющего эти прецеденты, при отсутствии какой-либо другой рабочей нагрузки для тестирующей системы."

"Performance Test. The performance test for the system will focus on measuring response time for use cases 2, 4, and 8 - 10. For these tests, a workload of one actor, executing these use cases without any other workload on the test system will be used."

" Тест Конфигурации. Тест Конфигурации будет реализован для определения и оценки поведения  цели теста в трех различных конфигурациях, сравнивая параметры производительности с таковыми в нашей эталонной конфигурации."

"Configuration Test. Configuration testing will be implemented to identify and evaluate the behavior of the target-of-test on three different configurations, comparing the performance characteristics to our benchmark configuration."

Техника

Technique

Техника должна описывать, как будет реализовано и выполнено тестирование. Включая то, что будет тестировано, основные действия, которые будут приняты в процессе выполнения теста, и метод(ы) , которые будут использованы для оценки результатов.

The technique should describe how testing will be implemented and executed. Include what will be tested, the major actions to be taken during test execution, and the method(s) used to evaluate the results.

Пример:

Example:

Функциональный Тест:

Functional Test:

  • Для каждого прецедентного потока событий будет определен репрезентативный набор сделок, каждая из которых представляет действия предпринятые актером при выполнении прецедента.
  • Минимум два прецедента будут разработаны для каждой сделки; один прецедент, чтобы отразить положительное состояние и один для отражения отрицательного (недопустимого) состояния.
  • В первой итерации, прецеденты 1 - 4, и 12 будут тестированы,  следующим образом:
    • Прецедент 1:
      • Прецедент 1 начинается с актера, уже зарегистрированного в приложении и в основном окне, и заканчивается, когда пользователь выбирает команду СОХРАНИТЬ.
      • Каждая тест-установка будет реализована и выполнена с  использованием Rational Robot.
      • Проверка и оценка выполнения каждой тест-установки будут выполнены с использованием следующих методов:
        • Выполнение тест-сценария (успешно ли выполняется каждый тест-сценарий, так как хотелось?)
        • Методы проверки Существования Окна или Данных Объекта (реализованных в тест-сценариях) будут использоваться для проверки того, что при выполнении теста ключевые окна отображаются и что указанные данные  захвачены/отображены  в цели теста.
        • База данных цели тестов (используется Microsoft Access) будет проверена до и после теста, чтобы удостовериться, что изменения, выполненные в процессе теста точно отражены в данных.
  • For each use case flow of events, a representative set of transactions will identified, each representing the actions taken by the actor when the use case is executed.
  • A minimum of two test cases will be developed for each transaction; one test case to reflect the positive condition and one to reflect the negative (unacceptable) condition.
  • In the first iteration, use cases 1 - 4, and 12 will be tested, in the following manner:
    • Use Case 1:
      • Use Case 1 begins with the actor already logged into the application and at the main window, and terminates when the user has specified SAVE.
      • Each test case will be implemented and executed using Rational Robot.
      • Verification and assessment of execution for each test case will be done using the following methods:
        • Test script execution (did each test script execute successfully and as desired?)
        • Window Existence, or Object Data verification methods (implemented in the test scripts) will be used to verify that key windows display and specified data is captured / displayed by the target-of-test during test execution.
        • The target-of-test's database (using Microsoft Access) will be examined before the test and again after the test to verify that the changes executed during the test are accurately reflected in the data.

Тест Производительности:

Performance Test:

  • Для каждого прецедента репрезентативный набор сделок, определенных в документе анализа рабочей нагрузки, будет реализован и выполнен с  использованием Rational Performance Studio (vu? сценарии) и Rational Robot (сценарии графического интерфейса пользователя (GUI)).
  • По крайней мере три рабочих нагрузки будут отражены в тест-сценариях, и при выполнении теста планируется следующее:
    • Нагрузочный объем работ: 750 пользователей (менеджеры 15%, продажи 50%, маркетинг 35%)
    • Пиковая рабочая нагрузка: 350 пользователей (менеджеры 10%, продажи 60%, маркетинг 30%)
    • Номинальная рабочая нагрузка: 150 пользователей (менеджеры 2%, продажи 75%, маркетинг 23%)
  • Тест-сценарии, используемые для выполнения каждой сделки, будут иметь соответствующие таймеры для определения времен ответа, таких как время всей сделки (как определено в документе анализа объема работ), и время действия ключевой сделки или времена процесса.
  • Тест-сценарии выполнят объемы работ в течение одного часа (если в документе анализа объема работ не выбрано другое).
  • Проверка и оценка выполнения для каждого теста (объема работ) будут включать:
    • Выполнение теста будет контролироваться с использованием гистограмм состояния (чтобы проверить, что тест и объемы работ выполняются как ожидалось и как хотелось)
    • Выполнение тест-сценария (каждый ли тест-сценарий выполняется успешно и так как хотелось?)
    • Захват и оценка определенных времен ответа  использует следующие сообщения:
      • Перцентиль Производительности
      • Время Ответа
  • For each use case, a representative set of transactions, as identified in the workload analysis document will be implemented and executed using Rational Performance Studio (vu scripts) and Rational Robot (GUI scripts).
  • At least three workloads will be reflected in the test scripts and test execution schedules including the following:
    • Stressed workload: 750 users (15 % managers, 50 % sales, 35 % marketing)
    • Peak workload: 350 users (10 % managers, 60 % sales, 30 % marketing)
    • Nominal workload: 150 users (2 % managers, 75% sales, 23 % marketing)
  • Test scripts used to execute each transaction will include the appropriate timers to capture response times, such as total transaction time (as defined in the workload analysis document), and key transaction activity or process times.
  • The test scripts will execute the workloads for one hour (unless noted differently by the workload analysis document).
  • Verification and assessment of execution for each test execution (of a workload) will include:
    • Test execution will be monitored using state histograms (to verify that the test and workloads are executing as expected and desired)
    • Test script execution (did each test script execute successfully and as desired?)
    • Capture and evaluation of the identified response times using the following reports:
      • Performance Percentile
      • Response Time

 

Критерии Завершения

Completion Criteria

Определение Критериев Завершения имеет две цели

Completion criteria are stated to for two purposes:

  • Определить приемлемое качество изделия
  • Определить когда объем работ теста был успешно реализован
  • identify acceptable product quality
  • identify when the test effort has been successfully implemented

Ясное формулирование критериев завершения должно включать следующие элементы:

A clear statement of completion criteria should include the following items:

  • Функция, поведение, или измеряемое условие
  • Метод измерения
  • Критерии или степень соответствия измерению
  • function, behavior, or condition being measured
  • method of measurement
  • criteria or degree of conformance to measurement

Пример 1

Example 1

  • Все запланированные тест-установки были выполнены
  • Все выявленные дефекты были переведены в согласованные на основании решения
  • Все запланированные тест-установки были повторно выполнены, и все известные дефекты были переведены в согласованные, и никаких новых дефектов обнаружено не было
  • All planned test cases have been executed
  • All identified defects have been addressed to an agreed upon resolution
  • All planned test cases have been re-executed and all known defects have been addressed as agreed upon, and no new defects have been discovered

Пример 2

Example 2

  • Все Тест-установки высокого приоритета были выполнены.
  • Все выявленные дефекты были переведены в согласованные на основании решения.
  • Вся Серьезность 1 или 2 дефекта была разрешена (статус = установленный или отложенный).
  • Все тест-установки высокого приоритета были повторно выполнены, и все известные дефекты переведены в согласованные, и никаких новых дефектов обнаружено не было.
  • All high priority test cases have been executed.
  • All identified defects have been addressed to an agreed upon resolution.
  • All Severity 1 or 2 defects have been resolved (status = fixed or postponed).
  • All high priority test cases have been re-executed and all known defects have addressed as agreed upon, and no new defects have been discovered.

Пример 3

Example 3

  • Все запланированные тест-установки были выполнены.
  • Все выявленные дефекты были переведены в согласованные на основании решения.
  • Вся Серьезность 1 или 2 дефекта была разрешена (статус = проверенный или отложенный).
  • Все тест-установки высокого приоритета были повторно выполнены, и все известные дефекты были переведены в согласованные, и никаких новых дефектов обнаружено не было.
  • All planned test cases have been executed.
  • All identified defects have been addressed to an agreed upon resolution.
  • All Severity 1 or 2 defects have been resolved (status = verified or postponed).
  • All high priority test cases have been re-executed and all known defects have addressed as agreed upon, and no new defects have been discovered.

Специальные Соображения

Special Considerations  

Этот раздел должен определить любые влияния или зависимости, которые могут, воздействие или влиять на объем работ теста, описанном в стратегии теста. Влияния могли бы включать:

This section should identify any influences or dependencies which may impact or influence the test effort describe in the test strategy. Influences might include:

  • Человеческие ресурсы (такие как доступность или необходимость в не связанных с тестом ресурсах для поддержки/участия в тесте)
  • Ограничения, (такие как ограничения оборудования или доступность, или потребности/недостаток специального оборудования)
  • Специальные требования, такие как планирование теста или доступ к системам
  • human resources (such as availability or need for non-test resources to support / participate in test)
  • constraints, (such as equipment limitations or availability, or the need / lack of special equipment)
  • special requirements, such as test scheduling or access to systems

Примеры:

Examples:

  • Базы данных тестов будут требовать поддержки разработчика/администратора базы данных для создания, модернизации и обновления данных теста.
  • Тестирование  производительности системы будет использовать серверы в существующей сети (которая поддерживает не связанный с тестом трафик). Тестирование будет нуждаться в планировании через какое-то время, чтобы гарантировать, что не будет никакого не связанного с тестом трафика в сети.
  • Цель теста должна согласовывать по времени систему наследования (или имитировать синхронизацию) для полнофункционального тестирования, которое  в последствие будет реализовано и выполнено
  • Test databases will require the support of a database designer / administrator to create, update, and refresh test data.
  • System performance testing will use the servers on the existing network (which supports non-test traffic). Testing will need to be scheduled after hours to ensure no non-test traffic on the network.
  • The target-of-test must synchronize the legacy system (or synchronization simulated) for full functional testing to be implemented and executed
